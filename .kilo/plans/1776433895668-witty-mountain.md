## Social Login Integration – Frontend Only

1. **UI additions**
   - Add social login buttons (Google, Facebook, Apple) to `login.html` with appropriate classes.
   - Include provider SDK scripts in the page head (e.g., Google Identity Services, Facebook SDK, Apple Sign‑In).
   - Create a minimal UI wrapper `handle<Provider>Login()` that obtains the provider token/credential.

2. **Auth wrapper**
   - Extend `window.MutsAuth` in `js/auth.js` with a `socialLogin(provider, idToken)` method that sends:
     ```js
     return fetchFromAPI(`/auth/social/${provider}`, {
       method: 'POST',
       body: { idToken }
     })
     .then(r => {
       if (r.success) setSession(r.user, false, { token: r.token, refreshToken: r.refreshToken });
       return r;
     });
     ```
   - Fallback: if API not ready, resolve with `{ success: false, message: 'Social login unavailable on static host' }`.

3. **Provider flow**
   - **Google**: Use `google.accounts.id.initialize` with client_id, callback calls `MutsAuth.socialLogin('google', credential)`.
   - **Facebook**: Load FB SDK, call `FB.login` to get access token, then `MutsAuth.socialLogin('facebook', accessToken)`.
   - **Apple**: Load Apple JS, use `AppleID.auth.signIn()` to obtain `id_token`, then `MutsAuth.socialLogin('apple', idToken)`.

4. **Documentation updates** (in `docs/FRONTEND_INTEGRATION.md`)
   - Add a **Social Login** section under *User/Auth* listing required backend endpoints:
     - `POST /api/auth/social/google`
     - `POST /api/auth/social/facebook`
     - `POST /api/auth/social/apple`
   - Specify request body (`{ idToken: string }`) and response schema (`{ success, user, token, refreshToken }`).
   - Note required server‑side verification of provider tokens, user creation/linking, JWT issuance.
   - Mention that the frontend will still work on static hosts—login button will show a warning if API not available.

5. **Error handling & security**
   - Validate `provider` string before sending request.
   - Encode any UI messages.
   - Log warnings when static fallback is used.

6. **Testing plan**
   - Manual test each button on localhost with mock backend (use tools like `json-server` or a simple Express stub). Verify session storage updates.
   - Verify fallback behavior on static host (e.g., Vercel preview) shows appropriate message.

**Deliverables**: UI code changes, `socialLogin` method, documentation section, test checklist.
