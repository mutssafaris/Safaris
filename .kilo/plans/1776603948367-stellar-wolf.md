# Wishlist Sync Across Devices - Implementation Plan

## Project Overview
**Goal**: Enable users to save items (products, destinations, tours, hotels) to a wishlist that syncs across devices when logged in, with offline support.

## Current State Analysis
| Component | Storage | Sync | User |
|-----------|--------|------|------|
| `africasa.html` wishlist | `mutssafaris_wishlist` (localStorage) | None | Any visitor |
| `favorites.js` destinations | `muts_favorites` (localStorage) | Tries /api/favorites | Logged-in user |
| No unified wishlist service | N/A | N/A | N/A |
| No wishlist management page | N/A | N/A | N/A |

**Issues:**
1. Two separate implementations (wishlist vs favorites) with different storage keys
2. No unified data model - product IDs vs destination IDs stored differently
3. No cross-device sync - localStorage stays on single device
4. No user-authenticated wishlist API
5. No dedicated wishlist management UI

---

## Implementation Phases

### Phase 1: Unified Wishlist Service
- Create `js/wishlist-service.js` - unified service with methods:
  - `addItem(itemType, itemId)` - product/destination/tour/hotel
  - `removeItem(itemType, itemId)`
  - `getItems(itemType?)` - get filtered or all items
  - `isInWishlist(itemType, itemId)` - check if saved
  - `syncWithAPI()` - sync to server when logged in
  - `getFromAPI()` - fetch from server
- Storage keys: Use existing but consolidate:
  - `muts_wishlist` (unified array of `{type, id, addedAt}`)
- Mock data: Create `data/wishlist.json` for offline/demo

### Phase 2: Backend API Structure
- Add mock API endpoints (if backend exists, document for implementation):
  - `GET /api/wishlist` - fetch user's wishlist
  - `POST /api/wishlist` - add item
  - `DELETE /api/wishlist/:type/:id` - remove item
- Auth: Requires `muts_user_session` JWT
- Response format: `{ success: true, data: { items: [...] }, message: "..." }`

### Phase 3: UI Integration
- Update `africasa.html`:
  - Replace local wishlist logic with `MutsWishlistService`
- Update `favorites.js`:
  - Update to use unified `MutsWishlistService`
- Add wishlist UI to header nav (with badge count)
- Create `pages/dashboard/my-wishlist.html` - manage wishlist items with:
  - List view grouped by item type
  - Remove buttons
  - "Add to Cart" for products
  - "Book Now" for tours/destinations
  - Empty state with CTA to browse

### Phase 4: Cross-Device Sync
- On login: Fetch wishlist from API, merge with local (local-first for offline)
- On logout: Keep local copy, clear API cache
- Conflict resolution: Server wins (more recent `addedAt`)
- Sync triggers:
  - On page load (if logged in)
  - On add/remove (optimistic local, then async API)
  - On re-login

### Phase 5: Edge Cases
- Handle deleted items: Mark as unavailable, allow removal
- Handle unavailable items: Show "No longer available" badge
- Price changes: Show current vs saved price
- Offline mode: Queue changes, sync on reconnect

---

## Implementation Rules

1. **No lazy execution** - Each phase complete before starting next
2. **API-first design** - All service methods work offline, sync async
3. **Security by default** - No eval(), sanitize inputs
4. **Mobile-first** - Test on mobile viewport
5. **Accessibility** - WCAG 2.1 AA, keyboard nav, screenreader labels
6. **Performance** - Lazy load wishlist items, cache product data
